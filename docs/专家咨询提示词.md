## 一、核心痛点：实盘训练数据严重不足

归爻的进化算法目前处于"理论完整、数据饥渴"的状态：

### 现状

| 可用的训练数据 | 数量 | 对进化算法的贡献 |
|--------------|:----:|:--------------:|
| 210只个股日线 × 827天 | 168,000 行 | ✅ 历史回测用，但只覆盖1种市场风格 |
| 22只ETF日线 × 800天 | 17,600 行 | ✅ ETF轮动用，样本够但只能验证低频策略 |
| **实盘交易记录** | **仅 4 条持仓** | ❌ **几乎为零，无法用于进化** |
| 用户主观过滤信号 | 无记录 | ❌ 尚未启用影子账本对比 |
| 极端行情数据（2015股灾/熔断） | 0 | ❌ V3引擎有能力模拟，但缺样本 |

**结论：能用历史数据跑回测，但缺乏多样化的训练集来验证策略在"未见过的行情"下的表现。**

### 为什么历史回测不够

- 2023-2026 年的 A 股是单一样本（电风扇行情 + 结构性牛市），不代表未来
- 48组业务参数在4个WF窗口全部失效（夏普<1.5），说明量价因子在2023-2026的单一风格下难以自我进化
- **策略需要见过更多样的行情（2015股灾的暴跌、2020年的V型反弹、2024年初的微盘踩踏），才能在实盘中识别相似情形**

## 二、硬件约束（不可逾越的设计前提）

专家在提出扩充方案时，**必须**考虑以下硬件限制。这些不是"建议"，是代码已经写死的铁约束：

| 资源 | 本地（Win10，12GB RAM） | 云端（Ubuntu 24.04，3.6GB RAM） |
|------|:-------------------:|:--------------------------:|
| 可用内存 | < 8GB（减去系统占用） | < 2GB（多个服务已在运行） |
| Numba 线程 | NUMBA_NUM_THREADS=2 | NUMBA_NUM_THREADS=2 |
| 数据库 | 仅用 Parquet/SQLite | 禁 PostgreSQL/MySQL/Redis |
| 消息队列 | 无 | 无 |
| Polars 模式 | streaming=True | streaming=True |
| 云服务进程 | — | 已占用：wechat server, watchdog, uvicorn, memory bridge, mcp server, hermes gateway（约300MB） |

**任何需要 > 2GB 内存的方案在云端不可行。数据库/消息队列方案直接否决。**

## 三、现有体系可供扩充方案对接的接口

### 3.1 V3 Numba 引擎可以做的事

```
backtest_kernel_v3(
    prices,           ← 可以传"人造行情"的OHLCV进去
    signals,          ← 100-200组随机参数的信号矩阵
    atrs,             ← 随行情一起变
    limit_down,       ← 随行情一起变
    total_capital,    ← 固定100K
    cash,             ← 可写入
    ...
)
```

**关键能力**：V3引擎对输入数据**没有来源限制**。它不关心 OHLCV 是从 mootdx 来的还是人工生成的。只要数据是 (S,) float64 的 NumPy 数组，引擎就能跑。

### 3.2 现有数据格式

```python
# market_v5.parquet 的结构
{
    "code": str,      # 股票代码 "600519"
    "date": str,      # "2025-06-15"
    "open": float64,
    "high": float64,
    "low": float64,
    "close": float64,
    "volume": int64,
    "amount": float64
}
```

### 3.3 进化参数空间

```python
# 48 组业务参数（6动量 × 4波动 × 2反转）
mom_thresholds = [0.0, 0.01, 0.02, 0.03, 0.05, 0.07]   # 动量阈值
vol_thresholds = [0.01, 0.02, 0.03, 0.05]               # 波动上限
rev_thresholds = [-0.02, -0.05]                          # 反转阈值
```

### 3.4 进化评估指标

| 指标 | 当前权重 | 说明 |
|------|:------:|------|
| WF 通过率 | 最高 | 在多个独立窗口中都正收益 |
| 盈亏比 | 高 | 赚的÷亏的 > 2.5 |

| 夏普比率 | 中 | > 0.3 |

## 四、对专家的具体请求

### 请求一：模拟盘样本扩充方案

当前 210 只股票 × 827 天的日线数据是**唯一的历史样本**。需要一种方法在**不改代码、不换格式**的前提下，从这个单一样本中生成多样化的新样本。具体需要：

1. **保留数据结构**：新样本必须是相同的 Parquet 格式、相同的字段、相同的类型
2. **保留统计特性**：新样本的涨跌幅分布、波动率、相关性不能完全随机（否则引擎只过拟合噪声）
3. **引入多样性**：新样本需要包含 2015 股灾、2020 V型反弹、熔断等情形，不能只是 2023-2026 风格的重排
4. **生成规模**：至少 5,000 条模拟盘交易，让 48 组参数中至少 5-10 组通过 WF
5. **可重复性**：生成方法必须产生可重用的结果，不能每次生成的都不同

### 请求二：评估方案的技术可行性

请在以下约束下给出评估（Yes/No + 理由）：

- [ ] 方案可以在 8GB 内存内运行（本地）
- [ ] 方案可以输出可直接喂给 V3 引擎的 NumPy 数组
- [ ] 方案不需要新增数据库/消息队列/Web 框架
- [ ] 方案产生的样本与真实历史数据的统计分布偏差可控
- [ ] 方案可以与现有的 `engine/data_layer.py` 无缝对接（即输出直接在 main.py 的现有流程中被使用）

### 请求三：给 Agent 的执行指令

如果方案通过评审，请给出**可以直接发给 Agent 执行**的步骤清单。格式要求：

- 每步一个命令或一段代码
- 明确指出输入文件、输出文件、中间产物
- 如果需要在本地或云端分步执行，明确标注
- 包含验证步骤（跑完之后怎么确认"生成对了"）
