# 归爻 WF 验证框架升级专家咨询

## 背景

归爻 V5 当前 Walk-Forward 验证采用 4 段非重叠滚动窗口（训练2年/测试半年），覆盖 2023-2026。GPT 审计指出：窗口数量偏少，未按市场状态分层（牛/熊/震荡/极端），容易高估策略稳定性。

## 现状

| 参数 | 当前值 |
|------|--------|
| 窗口数量 | 4 段（非重叠） |
| 窗口划分 | 按时间切（每段训练2年/测试半年） |
| 通过标准 | 夏普>0.3 / 回撤<25% / 年化>5%（ETF）；盈亏比>2.5 / 夏普>0.2（个股） |
| 时间范围 | 2023-02 ~ 2026-07（827天） |

## 核心问题（不预设答案）

1. 窗口应该按时间切还是按市场状态切（牛/熊/震荡各出一个窗口）？还是两者结合？
2. 如果有状态切分，状态标签如何定义？用宏观脑输出（regime.py）的历史评分？还是外部市场标签？
3. 窗口重叠如何处理？重叠是否会导致信息泄露？
4. 极端行情（2024年初微盘踩踏、2015股灾等）是否需要单独作为"压力窗口"而非WF窗口？如果时间范围只有827天，2015年数据如何获取？
5. WF 的通过标准是否需要分层（牛市和熊市的夏普门坎应该不一样）？
6. 除了夏普/回撤/年化，GPT建议增加"参数敏感性"和"收益分布一致性"指标——哪些是必要的？

## 约束（不可逾越）

C1: 不改 V3 Numba 引擎的回测接口
C2: 内存 < 8GB（本地）、< 2GB（云端）
C3: 仅 numpy/numba/polars + SQLite/Parquet，不新增框架
C4: 新窗口方案必须与当前 `param_search.py` 的 WF 接口兼容

## 期望产出

1. 最优的WF窗口切分方案（明确窗口数量、划分方式、重叠规则）
2. 如果状态切分优于时间切分，如何获取所需的市场状态标签
3. 新增的验证指标及其计算方法
4. 极端行情窗口的数据需求（如果827天覆盖不足，建议的数据来源）
5. 从当前方案迁移到新方案的增量代码量估算

回复格式：直接给出依次排列的思考过程、方案、和工程实施指引。不必反复确认。"无需等待确认，按最优方案给出。"
